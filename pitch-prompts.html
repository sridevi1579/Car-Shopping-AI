PROMPT 1 Architecture
One agent or two?
First decision — shaped the entire system. Separate agents mean separate system prompts, separate session histories, and clean isolation between concerns.
prompt.txt
Building a car shopping AI for a big company . Users chat naturally to find cars from a Database, get trade-in estimates, and see financing options. Stack: Flask, Next.js, Claude API. Should car search and trade-in be one agent or two completely separate ones with their own system prompts and session histories? And should the search/estimation logic be Python functions inside the agent or MCP tools Claude calls? Want the AI layer clean, separate from business logic.

PROMPT 5 Infrastructure
Persistent MCP connection, not per-call
Spawning a new MCP session per tool call added too much latency. One shared session held open at startup, serialized with a threading lock.
prompt.txt
Set up a FastMCP server on port over SSE with both tools registered — search_cars and estimate_trade_in. Flask connects once at startup and keeps the connection alive permanently — I tried spawning a new session per tool call and the latency was bad. Both agents share the same MCP session so needs a threading.Lock to serialise calls. Connection should retry up to 30 times with 1 second sleep in case the MCP server is still starting when Flask comes up.
⚑  Ensure the implementation is correct, the existing flow is preserved, and nothing breaks with the rest of the system.

PROMPT 6 Core Logic
The agent loop — this while loop is the system
Load history → append message → call Claude → if tool_use, execute and loop → if end_turn, save and return. Everything else is plumbing around this.
prompt.txt
Write the car search agent loop. History stored in SQLite under key search:{session_id}. Flow: load history → append user message → call Claude with system prompt + tool definitions → if stop_reason is tool_use, call mcp_session.call_tool(name, input), append assistant content + tool_result, loop again → if end_turn, save history and return final text. MAX_AGENT_TURNS = 10 cap. Also capture the raw car dicts from search_cars separately so Flask can return them to the frontend alongside the text response.

PROMPT 7 Search Tool
Scoring results, not just filtering them
Filter narrows the set. Score ranks it. Price 50%, mileage 30%, year 20% — normalised across results. External API calls only on the top 3, not all 20.
prompt.txt
My cars.csv columns: make, model, year, trim, price, mileage, condition, color, drive_type, location, vin, source. Write search_cars filtering by make, model, year (single or range like 2018-2022), budget, max_mileage, condition, location. Score results — price 50%, mileage 30%, year 20%, normalised across filtered set. Return top 20 sorted by score. Only enrich top 3 with NHTSA safety + fuel economy — don't want external API calls for all 20 results.
⚑  Ensure the implementation is correct, the existing flow is preserved, and nothing breaks with the rest of the system.

PROMPT 8 Bug Fix
Claude's API enforces tool message pairing
Naively trimming to last N messages split a tool_use from its tool_result across the cut boundary. The API rejects that. Fix: scan backward and always keep pairs intact.
prompt.txt
Agent throws `messages: tool_use and tool_result ids must match and appear in pairs` when I trim history to last 8 messages. My trim is splitting tool_use from its tool_result across the cut boundary — how do I always keep them as a pair?
