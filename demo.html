Multi-Agent Design
Two focused agents — each with its own tools, system prompt, and session namespace
① Car Search Agent
  • File: agents/carsearch_agent.py
  • Endpoint: POST /chat
  • Tools: search_cars, estimate_trade_in
  • Collects: make, model, year, budget, mileage via conversation
  • Enriches: top 3 results with NHTSA safety + fuel economy.gov
  • Session key: search:{session_id}
② Trade-In Agent
  • File: agents/tradein_agent.py
  • Endpoint: POST /estimate
  • Tool: estimate_trade_in
  • Collects: make, model, year, mileage, condition
  • Outputs: trade-in value + monthly payment breakdown
  • Session key: tradein:{session_id}
Why Two Agents?
Car search and trade-in are completely separate conversations — different goals, different tools, different rules Each agent has a focused system prompt — it only knows its job Easier to debug and change one without touching the other

Key Technical Decisions
Engineering choices made to keep the system reliable and cost-efficient
MCP — Persistent Connection
  • One SSE connection for the whole Flask lifetime
  • Thread-safe lock serialises tool calls
  • Not a subprocess per request — saves latency
Deal Scoring Formula
  • Normalised across all filtered results
  • Price 50% · Mileage 30% · Year 20%
  • Returns top 20, enriches only top 3 (API cost control)
History Trimming
  • Keeps last 8 turns (search) / 6 turns (trade-in)
  • Always preserves tool_use + tool_result pairs
  • Claude API requires chronological order — never reorders
Session Store
  • SQLite WAL mode — safe concurrent reads
  • 24h TTL — auto-expires on read
  • Namespaced keys prevent agents seeing each other's history
Trade-In Valuation
  • Base price from real DataBase market data (not hardcoded)
  • Condition multiplier from trade_in_rates table
  • Mileage adj: ±2% per 10k miles from expected wear
Core Agentic Loop
  • Claude returns tool_use — agent executes via MCP
  • Result fed back as tool_result — Claude responds
  • Loop repeats until Claude returns final text (end_turn)
  • Safety cap: MAX_AGENT_TURNS = 10 prevents infinite loops

Things I Got Wrong First
Design decisions I made, realised were wrong, and fixed — part of the build process
MCP Session Per Tool Call → Persistent Connection
  • First version: created a new MCP session on every single tool call — full connection handshake each time
  • Added latency on every request and made the code harder to manage
  • Fixed: one persistent SSE connection created at Flask startup, shared across all requests with a thread-safe lock — much faster
Each Agent on a Separate Port → Single MCP Server
  • First version: each agent had its own MCP server running on a different port — search on one, trade-in on another
  • Had to manage multiple processes, multiple connections, and more config
  • Fixed: one FastMCP server on port 5002 with both tools registered — simpler, one connection, one process
Car Search & Trade-In on Separate Ports → One Frontend
  • First version: car search UI and trade-in UI ran as separate frontends on different ports — two Next.js apps
  • Felt disconnected, users had to navigate between urls manually, no shared state
  • Fixed: single Next.js app with proper routing — car search, trade-in, and summary all as pages under one port
